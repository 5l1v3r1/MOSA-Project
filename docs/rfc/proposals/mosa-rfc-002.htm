Mosa-RFC-0002: MOSA License Affilliation

This specification provides a means for MOSA members to allow other MOSA members to use or port, and relicense certain portions of code.

This is a proposal for a new standard that is currently under review by the editor. It may or may not become a draft.

This standard has no implementation. It provides information on other standards and how to use them appropriately.

1. Introduction

Bodies within MOSA may wish to license their code in such a way that it may be (one or more of):

  • Ported
  • Used directly
  • Dynamically Linked
  • Statically Linked

In such a way that all of the following conditions are met:

  • The license may be changed to the new project's license of choice
  • Recognition will be given to all authors
  • Applicable changes are submitted to the original project

2. Business Rules

A project that wants to share code should add this code header to applicable files. This header MUST be placed below an OSI-approved license header.

Code Example: MOSA License Header

      /* This project is affiliated with MOSA (Managed Operating Systems Alliance).
       * Please see the mosalicense.txt for information on the license.
       *
       * If this file is copied place relevant information at the end of the authors 
       * section below according to the format:
       * < First Name > < Family Name >:
       *   < Project > - < License Name >
       *
       * Authors:
       */

The following file MUST be included in the repository in an obvious location, and MUST not be modified unless this document is ammended.

Code Example: mosalicense.txt

      This project is affiliated with MOSA (Managed Operating Systems Alliance).
      Other projects within MOSA may code files within this code repository that
      have the MOSA license header. They are allowed to use the files containing
      the header in the following ways:
      
      o Port it directly to another programming language,
      o Use it directly,
      o Compile against it in a dynamically linked fashion,
      o Compile against it in a statically linked fashion,

      without the former licensing provided that they:

      o Only append author, project and licensing information to the header.
      o Provide feedback and patches to the original project (for any portable
        changes).
      o Place the header in their code, below their license agreement.
      o Place a new OSI-approved license header in the code file.
        o This license must idemnify all contributors to the code from all
          liabilities.
      
      This exception is bound to the specific version of the file, that version of that
      file is known as affiliated. If the header is removed from the affiliated file, 
      the new version of the file becomes known as removed. All projects that use
      the affiliated file may continue to do so: but are not required to submit
      updates to the originating project. If the originating project re-adds the
      header to the removed file and wishes to obtain updates it must inform all the 
      other projects within MOSA of this action.
      
      Destination projects will not be held responsible for not providing updates
      to a previously removed file if they were not made aware that it became affiliated
      once again.
